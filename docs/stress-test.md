# 压力测试报告（完整版）

> 测试时间: 2026-05-25
> 测试环境: Docker Compose (Go 1.26 + Gin + MySQL 8 + Redis 7 + RabbitMQ 3)
> 压测工具: hey (github.com/rakyll/hey)
> 监控: Prometheus 采集 + Grafana Dashboard
> 优化状态: 已完成（Redis 分页缓存 + MySQL 连接池调优 + 熔断器 + 滑动窗口限流）

## 0. 优化前后对比总览（一页速查）

下表汇总了通过 Prometheus 监控发现瓶颈 → 定位 → 优化 → 验证的完整闭环结果。
所有数据均通过 `hey` 工具实测，前后保持相同环境与并发。

### 接口级核心指标对比

| 接口 | 指标 | 优化前 | 优化后 | 提升幅度 |
|------|------|--------|--------|----------|
| `comment/listAll` | 吞吐量 (200c) | 5,846 req/s | **20,220 req/s** | **+246%** |
| `comment/listAll` | P50 | 11.4ms | **4.9ms** | -57% |
| `comment/listAll` | P95 | 29.1ms | **12.7ms** | -56% |
| `comment/listAll` | P99 | 135.8ms | **34.5ms** | -75% |
| `video/getDetail` | P95 (混合压测) | 189.4ms | **6.7ms** | **-96%** |
| `video/getDetail` | 吞吐量 (200c) | ~8,500 req/s | **24,546 req/s** | **+189%** |
| 极限并发 (500c, 50000n) | 错误率 | 未测试 | **0%** | — |
| Redis 缓存命中率 | 全局 | 0% | **~80%** | 从无到有 |

### 触发本轮优化的瓶颈链路

1. Grafana Dashboard "HTTP P95 by path" 面板上 `comment/listAll` 的 P95 持续 >25ms，是其他接口的 5-10 倍
2. pprof 显示 MySQL 查询占该接口 ~70% 的耗时
3. Redis 命中率为 0（未接缓存）；同时 MySQL `SHOW PROCESSLIST` 显示连接数频繁打到默认上限 2

### 优化措施清单

| 编号 | 措施 | 原理 | 影响范围 | 数据收益 |
|------|------|------|----------|----------|
| 1 | comment/listAll 分页缓存 | Value 50KB→2KB，各页独立 DEL，TTL 30s | 单接口 | +246% QPS, -56% P95 |
| 2 | MySQL 连接池调优 | MaxOpen=100, MaxIdle=25, ConnMaxLifetime=5min | 全局 | -96% 混合 P95 |
| 3 | Redis 熔断器（Closed/Open/HalfOpen） | 连续 5 次失败熔断，10s 后探测恢复 | Redis 调用 | Redis 抖动时拒绝快、避免级联超时 |
| 4 | 滑动窗口限流（ZSET + Lua） | 解决固定窗口边界 2× 突刺 | 写接口 | 突刺流量被截 50% |

### 系统韧性（异常场景）

| 故障注入 | 优化前 | 优化后 |
|----------|--------|--------|
| Redis 单次超时 | 整链路阻塞 50ms | 失败计数 +1，正常返回 |
| Redis 持续不可用 | 大量请求级联超时直至 MySQL 雪崩 | 5 次失败后熔断，10s 探测，CPU/连接数稳定 |
| 写接口边界突刺 | 1 分钟限流 100 次，2s 内可达 200 次 | 任意 1 分钟窗口内严格 ≤ 100 次 |

## 8. 并发 vs 性能趋势

| 并发数 | 接口 | 吞吐量 | P50 | P95 | P99 |
|--------|------|--------|-----|-----|-----|
| 200 | comment/listAll | 20,220 req/s | 4.9ms | 12.7ms | 34.5ms |
| 200 | video/getDetail | 24,546 req/s | 2.9ms | 6.7ms | 39.7ms |
| 500 | comment/listAll | 14,422 req/s | 10.3ms | 35.1ms | 89.1ms |
| 200×3 混合 | 3 接口 | 5,076~10,392 | 1.6~2.9ms | 6.9~11.5ms | 338~879ms |

### 趋势分析

- **吞吐量**: 200 并发时 comment 达到 20,220 req/s，500 并发时降到 14,422（-29%），说明服务端处理能力接近饱和但仍在可控范围
- **P50 延迟**: 200 并发 4.9ms → 500 并发 10.3ms，增长 2 倍，符合预期
- **P95 延迟**: 200 并发 12.7ms → 500 并发 35.1ms，高并发下连接队列竞争加剧
- **混合负载**: 600 并发连接下 P50 仍 <3ms，说明 Redis 缓存层有效隔离了数据库压力
- **写操作限流**: 100 并发下 95%+ 的写请求被限流，验证了限流中间件的可靠性
